## Escopo entregue

### Instruções — `programs/valocracy/src/instructions/`

**`revoke.rs`** — `revoke(token_id)` Governor-only:
1. `has_one = governor` verifica autoridade
2. Lê `TokenOwner(token_id)` → obtém `owner` e `valor_id`
3. Lê `Valor(valor_id)` → calcula `effective_rarity(primary_track_id, valor)`
4. `user_stats.credential_level -= effective_rarity` (saturating)
5. Fecha `TokenOwner` e `TokenValorId` via Anchor `close = governor` (rent devolvido)
6. Emite `RevokeEvent { owner, token_id, valor_id, new_level }`

**`set_verified.rs`** — `set_verified(member, verified: bool)` Governor-only:
1. `has_one = governor` verifica autoridade
2. `user_stats.verified = verified`
3. Emite `VerificationChangedEvent { member, verified }`

### Wiring

- `instructions/mod.rs`: `pub mod revoke`, `pub mod set_verified` + re-exports
- `lib.rs`: entradas `revoke` e `set_verified` no `#[program]`

## Critério de aceite (do PRD)

- [x] Após revoke, `UserStats.credential_level` decrementado por `effective_rarity`
- [x] PDAs do token revogado fechadas (rent retornado)

## Caveats / dívidas técnicas

- `total_supply` não é decrementado em `revoke` — token_ids são permanentes (sem reuso). Paridade com Stellar.
- `permanent_level` não é decrementado: em v2-Solana sempre é 0 (DT-10 D1). O PRD menciona o campo mas não há caso prático.
- Token revogado não pode ser re-mintado no mesmo `token_id` (PDA fechada, mas `total_supply` não recua). Qualquer novo mint usa `total_supply + 1`.
