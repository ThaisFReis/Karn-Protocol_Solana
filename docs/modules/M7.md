### Fluxo de `credit_activity` (3 guards em ordem)

1. **Guard 1 — Circuit breaker:** `config.credit_paused == false`, senão `ActivityCreditPaused`
2. **Guard 2 — Track auth:** `credit_authority.track_ids.contains(&track_id)`, senão `CreditAuthorityUnauthorized`
3. **Guard 3 — Cap rolante:** Se `now >= period_start + 30d`, reseta `{credits=0, period_start=now}`; `effective = min(amount, 200 - window.credits)`
4. Atualiza `window.credits += effective`
5. Atualiza `user_stats.activity_level += effective`; `activity_expiry = now + 90d`
6. Emite `ActivityCreditedEvent { account, track_id, amount, effective_amount }`

## Caveats / dívidas técnicas

- `activity_level` acumula sem considerar o valor já decaído — comportamento intencional (paridade Stellar). Cada novo crédito reseta `activity_expiry = now + 90d` para o nível acumulado total.
- `credit_window` persiste após o período expirar (bump armazenado); reset feito no próximo `credit_activity`, não por cron.
- Não há limite no número de `CreditWindow` PDAs abertas (uma por conta credenciada). Accounts ficam abertas permanentemente pagando rent.
